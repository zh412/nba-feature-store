# NBA Feature Store Data Model

This document describes the warehouse schema used by the NBA Feature Store pipeline.

The warehouse follows a simplified **analytics star-schema design** that separates stable reference metadata from high-volume game statistics.

The system contains three primary tables:

- `pr_see_player_dimension`
- `pr_see_team_arena_dimension`
- `pr_see_daily_player_game_log`

The dimension tables store metadata that does not change on a per-game basis, while the feature store table contains player-level game statistics generated during ingestion.

---

# Player Dimension Table

`pr_see_player_dimension`

The player dimension table stores stable player attributes that do not change on a per-game basis.

Each row represents a **single NBA player**.

This table allows player metadata to be maintained separately from the high-volume game statistics table.

## Columns

| Column | Description |
|------|-------------|
| PLAYER_ID | Unique NBA player identifier |
| POSITION | Player's primary position |
| HEIGHT | Player height |
| EXP | Years of NBA experience |

## Data Sources

The dimension table is built automatically by the pipeline using the NBA Stats API endpoints:

- `commonallplayers`
- `commonplayerinfo`

The pipeline automatically creates and populates this table if it does not exist.

During ingestion, the dimension table is loaded and merged with player game logs to enrich each record with stable player attributes.

If the ingestion pipeline encounters a player not present in the dimension table (for example a newly signed player), the pipeline automatically retrieves metadata and inserts the player into the dimension before retrying ingestion.

---

# Team Arena Dimension Table

`pr_see_team_arena_dimension`

The team arena dimension table stores venue metadata for each NBA franchise.

Each row represents a **single NBA team**.

This table prevents repeated API requests for arena metadata and ensures consistent venue information across all games.

## Columns

| Column | Description |
|------|-------------|
| TEAM_ID | Unique NBA team identifier |
| TEAM_TRICODE | Three-letter team abbreviation |
| ARENA_NAME | Name of the team's home arena |
| ARENA_CITY | City where the arena is located |
| ARENA_STATE | State where the arena is located |

## Usage

During ingestion, the pipeline joins this table using:

`HOME_TEAM_ID`

Arena metadata is appended to each player game record to provide venue context.

---

# Player Game Feature Store

`pr_see_daily_player_game_log`

This is the **primary analytics table** produced by the pipeline.

Each row represents:

**one player in one NBA game**

The table is generated by merging multiple NBA Stats API endpoints and enriching the dataset with metadata from the dimension tables.

The final dataset contains **105 player-level features per NBA game**.

# Derived Game Context Features

The pipeline computes additional contextual features during ingestion.

These include:

- HOME_TEAM_POINTS
- AWAY_TEAM_POINTS
- POINT_MARGIN
- GAME_TOTAL_POINTS
- HOME_TEAM_WIN_FLAG
- PLAYER_TEAM_WIN_FLAG

These features describe the outcome and scoring context of the game.

---

# Warehouse Metadata

These columns are generated by the ingestion pipeline to support operational monitoring and idempotent loads.

- ROW_KEY
- INGESTED_AT_UTC
- LOAD_BATCH_ID

These fields allow the system to track ingestion batches, verify warehouse loads, and debug pipeline executions.

---

# Partitioning

The feature store table is partitioned by:

`GAME_DATE`

Partitioning allows efficient time-series queries and prevents full-table scans in BigQuery.

This significantly reduces query costs and improves performance when analyzing historical game data.

---

# Clustering

The table is clustered by:

- PLAYER_ID
- PLAYER_TEAM_ID

Clustering improves query performance for common analytical workloads such as:

- player performance analysis
- team-level aggregations
- matchup analysis
- time-series player trends

---

# Data Volume Characteristics

Each NBA game generates approximately:

- **20–30 player rows per team**
- **40–60 rows per game**

With **105 features per row**.

During the NBA season this results in a rapidly growing feature store optimized for analytical workloads.

---

# Table Relationships

The warehouse follows a simplified star schema:

```
                    pr_see_player_dimension
                            |
                            |
                            v
pr_see_team_arena_dimension ---> pr_see_daily_player_game_log
```

The dimension tables provide stable metadata that enriches player game records during ingestion.

The player dimension table provides player attributes such as position, height, and experience, while the team arena dimension table provides venue metadata such as arena name and location.

Both dimensions are joined during ingestion before the final dataset is written to the BigQuery feature store.

---

# Summary

The NBA Feature Store warehouse is designed for **analytics and modeling workflows**.

Key characteristics:

- Player-level feature store containing **105 features per game**
- Partitioned BigQuery table optimized for time-series queries
- Clustering designed for player and team level analytics
- Dimension tables separating stable metadata from high-volume game statistics
- Idempotent ingestion allowing safe historical reprocessing

This design provides a scalable analytics warehouse suitable for sports analytics research, feature engineering pipelines, and predictive modeling systems.
